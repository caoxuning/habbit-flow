# HabitFlow 中期检查说明文档

> 适用场景：软件综合课程设计中期检查  
> 项目名称：HabitFlow 自律目标追踪平台  
> 当前阶段：已完成个人自律闭环，并补充社交基础功能；同时新增 Java Spring Boot 三层架构复现版后端用于架构说明和后续迁移。

## 一、系统规划

### 1.1 项目背景

HabitFlow 面向学生和普通用户的自律目标管理场景。用户在学习、运动、阅读、英语、早睡、健身等目标执行过程中，常见问题包括：

- 目标记录分散，缺少统一管理入口。
- 每日执行情况难以长期追踪。
- 缺少连续坚持的正向反馈。
- 个人打卡缺少同伴监督和社区氛围。
- 数据统计不足，无法直观看到成长趋势。

因此，本系统规划为一个集 **账号体系、目标管理、每日打卡、补卡、统计图表、勋章奖励、好友社交、打卡圈子、社区动态** 于一体的 Web 平台。

### 1.2 系统定位

HabitFlow 不只是一个简单打卡工具，而是一个“个人自律 + 社交监督”的习惯养成平台。

系统分为两条主线：

| 主线 | 说明 | 当前完成度 |
|---|---|---|
| 个人自律闭环 | 用户注册登录、目标管理、每日打卡、补卡、记录查询、统计图表、勋章奖励 | 已基本完成 |
| 社交互动闭环 | 用户搜索、好友申请、好友列表、圈子列表、加入圈子、发布帖子、动态流 | 已完成基础功能，排行榜和消息提醒待增强 |

### 1.3 技术规划

当前项目采用前后端分离架构。

| 层次 | 当前实现 | 说明 |
|---|---|---|
| 前端 | Vue 3 + Vite + Element Plus + ECharts | 实现登录注册、侧边栏导航、目标、打卡、统计、勋章、社交页面 |
| 当前后端 | Python FastAPI + SQLAlchemy | 已完成主要业务接口，运行端口 8081 |
| Java 复现后端 | Spring Boot + Spring MVC + Spring Data JPA + MySQL | 新增在 `backend` 文件夹，用于按 Java 三层架构复现和说明系统逻辑 |
| 数据库 | MySQL | 存储用户、目标、打卡、勋章、好友、圈子、帖子等数据 |

中期检查时可以说明：项目当前可运行版本以后端 `backend_fastapi` 为主；同时已经将后端功能按 Java Spring Boot 三层架构在 `backend` 中复现，便于体现课程中常见的 Controller、Service、DAO 分层设计。

## 二、需求分析如何体现

需求分析不是只写功能列表，而是要体现“用户有什么问题、系统提供什么功能、每个功能如何被设计和实现”。本项目主要通过以下内容体现需求分析：

### 2.1 用户角色

| 用户角色 | 能做什么 |
|---|---|
| 未登录用户 | 注册、登录 |
| 登录用户 | 管理个人资料、创建目标、每日打卡、补卡、查看记录、查看统计、获得勋章、使用社交功能 |

### 2.2 功能需求拆分

系统需求已拆分为以下模块：

| 模块 | 需求说明 | 当前实现 |
|---|---|---|
| 用户账号 | 注册、登录、退出、个人信息、修改密码 | 已实现 |
| 目标管理 | 创建、编辑、归档目标、状态控制、完成率计算 | 已实现 |
| 打卡管理 | 今日打卡、快速打卡、补卡、记录查询、按目标和日期筛选 | 已实现 |
| 数据统计 | 目标总数、进行中目标、总打卡数、连续打卡、月度报表、完成率图表 | 已实现 |
| 勋章奖励 | 勋章规则、自动发放、我的勋章、获得时间展示 | 已实现 |
| 数据导出 | 打卡记录导出 Excel，支持目标和日期筛选 | 已实现 |
| 好友社交 | 用户搜索、好友申请、接受/拒绝、好友列表 | 已实现基础功能 |
| 打卡圈子 | 默认圈子、创建圈子、加入/退出、我的圈子 | 已实现基础功能 |
| 圈子帖子 | 圈内发帖、帖子列表、社区动态流 | 已实现基础功能 |
| 消息提醒 | 每日提醒、到期提醒、中断提醒、消息中心 | 待实现 |
| 排行榜 | 好友排行、圈子排行、目标类型排行 | 待实现 |

### 2.3 业务规则体现

需求分析中的规则已经落到了后端业务校验中：

| 业务规则 | 代码体现 |
|---|---|
| 用户名唯一 | 注册和修改资料时校验用户名是否重复 |
| 密码加密 | 使用 bcrypt 加密保存密码 |
| 登录鉴权 | 使用 token/JWT，前端请求自动携带 Authorization |
| 目标结束日期不能早于开始日期 | 创建/编辑目标时后端校验 |
| 只有进行中目标可以打卡 | 打卡接口校验目标状态为 `ACTIVE` |
| 同一目标同一天不能重复打卡 | 打卡前查询是否已有同日记录 |
| 补卡日期必须早于今天 | 补卡接口校验日期 |
| 补卡日期必须在目标周期内 | 统一打卡保存逻辑校验目标起止日期 |
| 打卡后自动发放勋章 | 打卡成功后调用勋章判断逻辑 |
| 归档目标保留历史记录 | 删除目标改为状态 `ARCHIVED`，记录仍保留 |
| 未加入圈子不能发帖 | 发帖接口检查圈子成员关系 |

### 2.4 需求追溯方式

可以用“需求 -> 接口 -> 页面 -> 数据表”的方式向老师说明。

示例：

| 需求 | 后端接口 | 前端页面 | 数据表 |
|---|---|---|---|
| 今日打卡 | `POST /api/check-ins` | 打卡管理页 | `check_in` |
| 创建目标 | `POST /api/goals` | 目标管理页 | `goal` |
| 我的勋章 | `GET /api/badges/mine` | 勋章奖励页 | `badge`, `user_badge` |
| 好友申请 | `POST /api/social/friend-requests` | 社交圈子页/好友管理 Tab | `friendship` |
| 圈子发帖 | `POST /api/social/circles/{id}/posts` | 社交圈子页/圈子广场 Tab | `circle_post` |

这就是需求分析在项目中的实际体现：每个功能不是孤立文字，而是有接口、有页面、有数据表、有业务规则。

## 三、系统设计

### 3.1 总体架构设计

系统采用前后端分离模式。

```text
用户浏览器
  |
  | Vue 3 前端页面
  | - Element Plus 组件
  | - ECharts 图表
  | - Axios API 封装
  v
后端 REST API
  |
  | 业务逻辑层
  | - 用户认证
  | - 目标管理
  | - 打卡校验
  | - 勋章发放
  | - 统计聚合
  | - 社交圈子
  v
MySQL 数据库
  |
  | user / goal / check_in / badge / user_badge
  | friendship / social_circle / circle_member / circle_post
```

### 3.2 前端设计

前端采用 Vue 3 单页应用结构，主要页面包括：

- 登录注册页
- 数据概览页
- 目标管理页
- 打卡管理和打卡记录页
- 成长日志页
- 勋章奖励页
- 社交圈子页
- 个人中心页

前端特点：

- 使用 `localStorage` 保存登录 token。
- Axios 请求拦截器自动携带 token。
- 401 失效时自动清除登录信息。
- 使用 Element Plus 表单、表格、弹窗、标签、日期选择器。
- 使用 ECharts 展示月度成长报表和目标完成率图表。

### 3.3 后端设计

后端按模块拆分接口：

| 模块 | 路由前缀 | 作用 |
|---|---|---|
| 认证模块 | `/api/auth` | 注册、登录 |
| 用户模块 | `/api/user` | 个人信息、修改密码 |
| 目标模块 | `/api/goals` | 目标 CRUD、归档、进度 |
| 打卡模块 | `/api/check-ins` | 打卡、补卡、记录筛选 |
| 统计模块 | `/api/stats` | 数据概览、成长日志 |
| 勋章模块 | `/api/badges` | 我的勋章 |
| 导出模块 | `/api/exports` | Excel 导出 |
| 社交模块 | `/api/social` | 好友、圈子、帖子、动态流 |

### 3.4 数据库设计

核心表如下：

| 表名 | 说明 |
|---|---|
| `user` | 用户账号 |
| `goal` | 用户目标 |
| `check_in` | 打卡记录 |
| `badge` | 勋章规则 |
| `user_badge` | 用户获得的勋章 |
| `friendship` | 好友申请和好友关系 |
| `social_circle` | 打卡圈子 |
| `circle_member` | 圈子成员 |
| `circle_post` | 圈子帖子 |

主要关系：

```text
user 1:N goal
user 1:N check_in
goal 1:N check_in
user N:M badge，通过 user_badge 实现
user 与 user 通过 friendship 建立好友关系
user N:M social_circle，通过 circle_member 实现
social_circle 1:N circle_post
user 1:N circle_post
```

## 四、后端架构如何体现

本项目当前保留了 FastAPI 后端，同时在 `backend` 目录中新增了 Java Spring Boot 三层架构复现版。Java 后端可以用来清晰说明后端架构设计。

### 4.1 Java 三层架构

```text
Controller 层
  接收 HTTP 请求、解析参数、返回统一响应
  例如 AuthController、GoalController、CheckInController

Service 层
  处理业务逻辑、校验规则、事务控制
  例如 GoalService、CheckInService、BadgeService、SocialService

DAO 数据层
  通过 Spring Data JPA 操作数据库
  例如 UserRepository、GoalRepository、CheckInRepository

Entity 实体层
  映射数据库表
  例如 User、Goal、CheckIn、Badge、Friendship、SocialCircle
```

### 4.2 以今日打卡为例说明逻辑

```text
前端点击“今日打卡”
  |
  v
CheckInController 接收 POST /api/check-ins
  |
  v
CheckInService.checkIn()
  |
  | 校验目标是否属于当前用户
  | 校验目标状态是否 ACTIVE
  | 校验打卡日期是否在目标周期内
  | 校验同一目标当天是否已打卡
  v
CheckInRepository 保存 check_in 记录
  |
  v
BadgeService 判断是否满足勋章条件
  |
  v
返回打卡结果、推荐内容、同路人寄语
```

### 4.3 以目标管理为例说明逻辑

```text
GoalController
  - GET /api/goals 查询目标列表
  - POST /api/goals 创建目标
  - PUT /api/goals/{id} 编辑目标
  - DELETE /api/goals/{id} 归档目标

GoalService
  - 校验日期
  - 判断目标归属
  - 计算完成率
  - 计算应完成次数
  - 计算连续打卡天数

GoalRepository
  - 查询当前用户目标
  - 查询非归档目标
  - 根据目标 id 和 userId 查询目标
```

### 4.4 以勋章为例说明逻辑

```text
CheckInService 保存打卡记录
  |
  v
evaluateBadges()
  |
  | 统计总打卡次数
  | 统计连续打卡天数
  v
BadgeService.grantIf()
  |
  | 判断是否满足勋章条件
  | 判断是否已获得过该勋章
  v
UserBadgeRepository 保存用户勋章
```

## 七、中期检查讲述建议

中期检查可以按以下顺序讲：

1. **系统规划**：为什么做 HabitFlow，目标用户是谁，系统分成个人自律和社交互动两条主线。
2. **需求分析**：从用户角色、功能模块、业务规则、需求追溯矩阵说明需求不是空泛描述，而是已经落到接口、页面和数据表。
3. **系统设计**：讲前后端分离、数据库表关系、接口模块划分。
4. **后端架构**：重点讲 Java Spring Boot 三层架构，Controller 接请求，Service 做业务，DAO 操作数据库，Entity 映射数据表。
5. **实现工作**：展示当前已经完成的注册登录、目标、打卡、补卡、统计、勋章、社交基础功能。
6. **项目亮点**：成长日志、内容推荐、Excel 导出、软删除归档、勋章激励、社交圈子。
7. **后续计划**：消息提醒、排行榜、点赞评论、测试和文档完善。
