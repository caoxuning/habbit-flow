# HabitFlow 三人功能分工安排

本文按《HabitFlow 功能设计清单》拆分三名成员的工作量，原则是每个人负责一条相对完整的功能链路，既方便开发，也方便后续答辩时说明个人贡献。

## 当前前端展示状态

前端当前可以正常启动和展示登录注册页，`npm.cmd run build` 已通过，开发服务器首页请求返回 `200`。

已实现页面和功能主要集中在第一阶段个人功能闭环：

- 登录注册页
- 数据概览页
- 目标管理页
- 打卡管理和打卡记录页
- 勋章奖励页
- 个人中心页

需要注意：当前项目实际后端为 `backend_fastapi`，默认运行端口为 `8081`，前端代理应指向 `http://localhost:8081`。如果后端没有启动，前端页面仍可展示，但登录、注册、目标、打卡、统计等接口功能无法正常联调。

尚未完整实现的功能：

- 消息提醒和消息中心
- 好友管理
- 打卡圈子
- 圈子帖子和社区动态
- 社交排行榜
- 评论、点赞、日历热力图等增强功能

## 成员一：个人账号与目标管理负责人

负责范围：用户与账号功能、目标管理功能、基础页面结构。

对应功能设计清单：

- 2.1 用户注册
- 2.2 用户登录
- 2.3 退出登录
- 2.4 个人信息维护
- 2.5 修改密码
- 3.1 创建目标
- 3.2 编辑目标
- 3.3 删除目标
- 3.4 目标状态管理
- 3.5 目标完成率计算
- 14 建议前端页面中的登录注册页、目标管理页、个人中心页

主要工作内容：

- 设计用户表、目标表相关字段。
- 完成注册、登录、JWT 校验、退出登录逻辑。
- 完成个人资料查看、修改资料、修改密码接口。
- 完成目标新增、查询、编辑、删除接口。
- 完成目标状态控制，包括进行中、已暂停、已完成。
- 前端实现登录注册、侧边栏导航、目标管理、个人中心页面。
- 和成员二约定目标数据结构，保证打卡功能可以直接使用目标列表。

答辩可说明贡献：

- 完成系统账号体系和权限访问基础。
- 完成目标管理 CRUD。
- 为后续打卡、统计、社交功能提供用户和目标数据基础。

## 成员二：打卡、统计与勋章负责人

负责范围：打卡管理、数据统计、勋章奖励、可视化图表。

对应功能设计清单：

- 4.1 今日打卡
- 4.2 快速打卡
- 4.3 补卡功能
- 4.4 打卡记录查看
- 4.5 按目标筛选记录
- 5.1 数据概览
- 5.2 月度成长报表
- 5.3 目标完成率图表
- 5.4 连续打卡统计
- 6.1 勋章规则
- 6.2 自动发放勋章
- 6.3 我的勋章
- 14 建议前端页面中的数据概览页、打卡管理页、打卡记录页、勋章奖励页

主要工作内容：

- 设计打卡记录表、勋章表、用户勋章表。
- 完成今日打卡接口，校验重复打卡和目标状态。
- 完成补卡接口，校验补卡日期、目标周期和重复补卡。
- 完成打卡记录查询，支持按目标筛选。
- 实现总完成次数、当前连续打卡天数、平均完成率、月度报表等统计逻辑。
- 实现勋章初始化、自动发放和我的勋章查询。
- 前端实现每日打卡、补卡、打卡记录、数据概览图表、勋章展示。
- 负责 ECharts 图表展示和页面数据刷新。

答辩可说明贡献：

- 完成个人自律闭环中的核心业务。
- 实现防重复打卡、补卡校验、连续打卡统计等关键算法。
- 实现统计图表和勋章激励机制，增强系统展示效果。

- 负责整体接口测试、联调测试、测试用例、课程设计文档和 PPT 材料整理。

- 完成好友、圈子、动态流等互动功能。
- 负责测试和文档，保证项目可演示、可说明、可交付。

## 推荐开发顺序

第一阶段先完成成员一和成员二的个人功能闭环：

- 注册登录
- 目标管理
- 今日打卡和补卡
- 打卡记录
- 数据概览
- 勋章奖励

第二阶段由成员三补齐社交基础闭环：

- 用户搜索
- 好友申请
- 好友列表
- 圈子列表
- 加入圈子
- 圈内发帖
- 社区动态流

第三阶段三人共同完善答辩亮点：

- 消息提醒
- 排行榜
- 日历热力图
- 评论点赞
- 测试用例
- 系统文档和 PPT

## 工作量对比

整体工作量建议按以下比例理解：

- 成员一：约 30%，负责账号、权限、目标管理和基础前端框架。
- 成员二：约 35%，负责打卡业务、统计算法、勋章规则和图表展示。
- 成员三：约 35%，负责社交互动、消息提醒、排行榜、测试和文档。

如果课程设计时间紧，可以把成员三的排行榜、评论点赞、日历热力图作为拓展功能，优先保证好友、圈子、动态流三个社交基础功能可演示。
