# HabitFlow 三人功能分工与讲解指引

更新时间：2026-06-25

本文档按当前 `main` 分支功能划分三名成员的负责范围，并给出答辩时的讲解顺序、页面入口、接口说明和演示重点。分工原则是：每个人负责一条相对完整的业务链路，讲解时既能说明自己做了什么，也能说明该模块如何接入整个 HabitFlow 系统。

## 1. 当前系统总览

HabitFlow 当前已经形成完整闭环：

1. 用户登录系统。
2. 创建目标并设置周期、日期和重要程度。
3. 目标进入工作台、任务日历和今日打卡。
4. 用户完成打卡并获得激励反馈。
5. 打卡记录进入统计、成长日志和勋章体系。
6. 用户可以把打卡分享到圈子或通知好友。
7. 好友、圈子、聊天、动态、排行榜和消息提醒共同构成社交监督闭环。

主要前端入口：

- 工作台：首页概览、今日打卡、最近记录、排行榜预览。
- 目标：任务日历、目标管理、打卡记录。
- 成长：成长日志、消息提醒、勋章奖励。
- 社交：好友管理、好友聊天、圈子广场、社区动态、打卡排行榜。
- 账号：个人信息和安全管理。

主要后端目录：

- `backend_fastapi/app/routers/auth.py`：登录注册。
- `backend_fastapi/app/routers/users.py`：个人资料和密码。
- `backend_fastapi/app/routers/goals.py`：目标管理。
- `backend_fastapi/app/routers/checkins.py`：打卡和补卡。
- `backend_fastapi/app/routers/stats.py`：统计和成长日志。
- `backend_fastapi/app/routers/badges.py`：勋章。
- `backend_fastapi/app/routers/notifications.py`：消息提醒。
- `backend_fastapi/app/routers/social.py`：好友、聊天、圈子、动态、排行榜、打卡分享。

## 2. 成员一：账号、目标与任务日历负责人

### 2.1 负责范围

成员一负责系统的基础身份和目标数据链路：

- 注册、登录、退出。
- 登录态保存和失效处理。
- 个人信息维护。
- 修改密码入口。
- 目标新增、编辑、删除、状态管理。
- 任务日历月视图和日期详情。

### 2.2 对应功能清单

- 1. 账号与登录。
- 3. 目标管理。
- 4. 任务日历。
- 16. 顶部导航与页面交互中的基础导航。
- 17.1 认证接口。
- 17.2 目标接口中的目标管理部分。

### 2.3 页面讲解入口

答辩时从这些页面讲：

1. 登录页：展示账号登录、注册入口和登录后跳转。
2. 顶部导航“目标”：进入目标管理。
3. 目标管理页：展示目标列表、新建目标、编辑目标、删除目标。
4. 任务日历页：展示不同重要程度目标在日历中的颜色区分。
5. 点击某一天：展示当天目标详情和完成状态。
6. 顶部导航“账号”：展示个人资料维护和安全入口。

### 2.4 后端接口说明

可重点说明这些接口：

- `POST /api/auth/register`：注册用户。
- `POST /api/auth/login`：登录并返回 token。
- `GET /api/user/profile`：获取当前用户信息。
- `PUT /api/user/profile`：修改用户名和邮箱。
- `PUT /api/user/password`：修改密码。
- `GET /api/goals`：查询目标列表。
- `POST /api/goals`：新建目标。
- `PUT /api/goals/{id}`：编辑目标。
- `DELETE /api/goals/{id}`：删除目标。

### 2.5 关键实现说明

- 目标是后续打卡、日历、统计和社交分享的基础数据。
- 目标包含名称、类型、周期、开始日期、结束日期、每日次数、重要程度和状态。
- 重要程度影响日历颜色：普通、重要、紧急分别用不同视觉样式展示。
- 新建目标后会写入数据库，并生成一条目标相关提醒，顶部未读数量可以刷新出来。
- 个人信息修改后同步更新顶部导航显示名称。

### 2.6 演示步骤建议

1. 登录系统。
2. 打开目标管理，新建一个“英语听力 20 分钟”目标。
3. 设置周期为每日，重要程度为重要。
4. 保存后回到任务日历，查看目标已出现在对应日期。
5. 点击日期，展示当天目标详情。
6. 进入账号页面，演示用户名或邮箱可维护。

### 2.7 成功标准

- 用户能登录并保持登录状态。
- 目标创建后可以在目标管理和日历中看到。
- 编辑目标后日历和打卡页面同步变化。
- 删除目标后不再出现在目标列表和日历中。
- 个人资料修改后顶部用户名同步更新。

## 3. 成员二：打卡、统计、成长与勋章负责人

### 3.1 负责范围

成员二负责个人自律闭环：

- 工作台数据概览。
- 今日打卡。
- 补卡。
- 打卡记录筛选和导出。
- 成长日志。
- 激励弹窗。
- 勋章奖励。
- 首页排行榜预览展示。

### 3.2 对应功能清单

- 2. 工作台首页。
- 5. 打卡记录。
- 7. 成长日志。
- 9. 勋章奖励。
- 14.3 工作台排行榜预览。
- 15.1 工作台统计。
- 17.2 打卡接口。
- 17.4 勋章接口。

### 3.3 页面讲解入口

答辩时从这些页面讲：

1. 工作台：展示今日完成率、今日打卡、连续天数、累计打卡。
2. 今日打卡区域：演示点击打卡。
3. 激励弹窗：展示随机激励语句和动效。
4. 打卡记录页：展示按目标、日期筛选。
5. 补卡区域：演示历史日期补卡规则。
6. 成长日志：展示打卡事件和勋章事件。
7. 勋章奖励：展示已获得勋章和未解锁规划。

### 3.4 后端接口说明

可重点说明这些接口：

- `GET /api/check-ins`：查询打卡记录。
- `POST /api/check-ins`：今日打卡。
- `POST /api/check-ins/makeup`：补卡。
- `GET /api/stats/dashboard`：工作台统计。
- `GET /api/stats/timeline`：成长日志时间线。
- `GET /api/badges/mine`：我的勋章。
- `GET /api/exports/checkins`：导出打卡记录。

### 3.5 关键实现说明

- 今日打卡会校验目标状态、目标周期和重复打卡。
- 补卡只能补历史日期，不能补未来日期。
- 打卡成功后进入统计、成长日志和勋章系统。
- 激励弹窗内容会随机变化，避免用户每天看到同一句话。
- 勋章根据累计打卡、连续打卡、目标类型和社交行为逐步解锁。
- 工作台只展示高频数据，完整记录放在打卡记录和成长日志中。

### 3.6 演示步骤建议

1. 在工作台查看今日目标和完成率。
2. 点击未完成目标的“去打卡”。
3. 查看打卡成功后的激励弹窗。
4. 进入打卡记录页，查看刚才的记录。
5. 进入成长日志，说明打卡事件被记录进时间线。
6. 进入勋章页，说明完成打卡后可能解锁勋章。

### 3.7 成功标准

- 今日打卡成功后今日完成数增加。
- 同一天同一目标不能重复打卡。
- 打卡记录可按目标和日期筛选。
- 成长日志能看到打卡事件。
- 勋章页面能展示已获得和未获得勋章。

## 4. 成员三：消息提醒、好友社交、圈子与排行榜负责人

### 4.1 负责范围

成员三负责社交监督和消息联动：

- 消息提醒分类展示。
- 未读提醒确认。
- 好友搜索。
- 好友申请、同意、拒绝。
- 好友今日打卡看板。
- 好友聊天。
- 圈子创建、加入、退出。
- 圈子动态和社区动态。
- 点赞、评论和鼓励。
- 打卡分享到圈子或好友。
- 好友排行榜和圈子排行榜。

### 4.2 对应功能清单

- 6. 打卡分享与社交联动。
- 8. 消息提醒。
- 10. 好友管理。
- 11. 好友聊天。
- 12. 圈子广场。
- 13. 社区动态。
- 14. 排行榜。
- 17.3 社交接口。
- 17.4 消息接口。

### 4.3 页面讲解入口

答辩时从这些页面讲：

1. 成长 - 消息提醒：展示左侧分类、未读数、确认按钮和今日建议。
2. 社交 - 好友管理：展示好友今日打卡看板、搜索用户、好友申请。
3. 社交 - 好友聊天：展示会话列表、消息内容、发送时间和输入框。
4. 社交 - 圈子广场：展示圈子列表、创建圈子、加入圈子、圈内动态。
5. 社交 - 社区动态：展示加入圈子的动态流和互动。
6. 社交 - 打卡排行榜：展示好友排行榜和圈子排行榜。
7. 打卡成功弹窗：展示分享到好友或圈子的入口。

### 4.4 后端接口说明

可重点说明这些接口：

- `GET /api/notifications`：查询提醒列表。
- `GET /api/notifications/unread-count`：查询未读数量。
- `PUT /api/notifications/{id}/read`：确认单条提醒。
- `PUT /api/notifications/read-all`：全部已读。
- `DELETE /api/notifications/{id}`：删除提醒。
- `GET /api/social/users/search`：搜索用户。
- `POST /api/social/friend-requests`：发送好友申请。
- `GET /api/social/friend-requests`：获取好友申请。
- `PUT /api/social/friend-requests/{id}/accept`：同意好友申请。
- `PUT /api/social/friend-requests/{id}/reject`：拒绝好友申请。
- `GET /api/social/friends`：好友列表。
- `GET /api/social/friends/checkins/today`：好友今日打卡看板。
- `GET /api/social/friends/{friend_id}/messages`：获取私信。
- `POST /api/social/friends/{friend_id}/messages`：发送私信。
- `GET /api/social/circles`：圈子列表。
- `POST /api/social/circles`：创建圈子。
- `POST /api/social/circles/{id}/join`：加入圈子。
- `DELETE /api/social/circles/{id}/leave`：退出圈子。
- `GET /api/social/circles/{id}/posts`：圈子动态。
- `POST /api/social/circles/{id}/posts`：发布动态。
- `POST /api/social/checkins/share`：分享打卡到圈子或好友。
- `GET /api/social/feed`：社区动态流。
- `GET /api/social/leaderboards`：好友和圈子排行榜。

### 4.5 关键实现说明

- 消息提醒不是页面刷新时临时生成，而是读取数据库中的提醒记录。
- 用户点击“确认”后，提醒变为已读，但历史消息保留。
- 顶部导航的未读提醒数量来自后端未读数接口。
- 好友聊天消息区分发送者，展示头像、姓名、发送时间和消息内容。
- 打卡分享到圈子时，动态关联 `check_in_id`，能追溯到具体打卡记录。
- 圈子排行榜统计的是圈子相关打卡分享，而不是简单统计用户所有打卡。
- 好友排行榜统计当前用户和好友的累计打卡次数。

### 4.6 演示步骤建议

1. 进入消息提醒，点击未读提醒的“确认”，说明未读红点减少但历史消息保留。
2. 进入好友管理，搜索一个用户并发送好友申请。
3. 切换到另一个账号或使用已有数据，展示同意/拒绝申请。
4. 进入好友聊天，发送一条鼓励消息。
5. 创建或加入一个圈子。
6. 完成一次打卡，在弹窗中选择分享到圈子或好友。
7. 进入圈子广场或社区动态，查看打卡分享动态。
8. 进入打卡排行榜，查看好友和圈子的打卡次数排序。

### 4.7 成功标准

- 未读提醒确认后不再计入未读数。
- 已确认提醒仍能在全部提醒中看到。
- 好友申请流程可以完成。
- 好友之间可以互相发送消息。
- 圈子可以创建、加入、退出和发布动态。
- 打卡分享能出现在对应圈子或好友私信中。
- 排行榜能按打卡次数排序展示。

## 5. 三人工作量对比

建议按以下比例说明工作量：

- 成员一：约 30%。负责账号体系、目标管理、任务日历和基础数据结构。
- 成员二：约 35%。负责打卡闭环、统计、成长日志、勋章和工作台数据展示。
- 成员三：约 35%。负责消息提醒、好友、聊天、圈子、社区动态、排行榜和社交联动。

## 6. 答辩整体讲解顺序

推荐按业务闭环讲，而不是按代码文件讲：

1. 成员一先讲账号登录和目标创建。
2. 成员一继续讲任务日历如何展示目标安排。
3. 成员二讲工作台如何汇总今日目标。
4. 成员二演示今日打卡、补卡、记录、成长日志和勋章。
5. 成员三接着讲打卡后的分享弹窗。
6. 成员三演示好友消息、圈子动态和社区互动。
7. 成员三讲消息提醒和排行榜如何促进持续打卡。
8. 三人共同说明远端数据库支持协作开发和持久化数据。

## 7. 联调与验收建议

### 7.1 启动检查

- 后端启动后确认 `http://localhost:8081` 可访问。
- 前端启动后确认 `http://localhost:5173` 可访问。
- 前端接口代理应指向后端 `8081`。
- 使用 RDS 时确认 `DATABASE_URL` 已设置。

### 7.2 基础验收链路

1. 注册或登录账号。
2. 创建目标。
3. 查看日历。
4. 完成打卡。
5. 查看工作台统计变化。
6. 查看成长日志。
7. 查看勋章。
8. 分享打卡到圈子或好友。
9. 查看社交动态和排行榜。
10. 查看消息提醒并确认未读。

### 7.3 协作注意事项

- 每个人开发前应从 `main` 拉新分支。
- 开发完成后发起 PR 合并到 `main`。
- 如果同一个函数冲突，先和负责该模块的人确认业务逻辑，再解决冲突。
- 本地包含数据库密码的启动脚本不要提交到 GitHub。
- 课程报告、截图、压缩包等本地文件不要随代码一起提交。
