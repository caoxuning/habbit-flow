## 二、你问到的技术概念解释

### 2.1 RAG（检索增强生成）是什么？

**全称**：Retrieval-Augmented Generation（检索增强生成）

**不是单一技术，而是架构模式**，它结合了"检索"和"生成"两步：

```
用户提问 "英语怎么学？"
       │
       ▼
┌───────────────────┐
│  第一步：检索      │
│  从知识库中寻找    │
│  最相关的内容片段  │ ← 这里用到向量数据库 / 文本检索
│  "如何背单词"      │
│  "阅读技巧"       │
└───────┬───────────┘
        │ 找到的参考内容
        ▼
┌───────────────────┐
│  第二步：生成      │
│  把参考内容+问题   │
│  一起发给大模型    │ ← 这里用到 LLM API
│  让 AI 基于资料    │
│  给出回答          │
└───────────────────┘
```

**在本项目中的应用场景**：
- 用户问"英语目标怎么达成"，系统先检索数据库中英语高完成率的用户的打卡记录和学习方法，再让 AI 总结成个性化建议

**RAG 和"直接调 API"的区别**：

| 对比项 | 直接调 API | RAG |
|:---|:---|:---|
| 回答来源 | 大模型自身知识 | 你提供的资料 + 大模型总结 |
| 数据时效性 | 训练数据截止时间 | 可以是最新的 |
| 内容可控性 | 可能胡编 | 基于真实资料生成 |
| 实现复杂度 | 1 个 API 调用 | 需建知识库 + 检索 + 生成 |
| 适用于本项目 | 简单建议 | 深度学习方法问答 |

### 2.2 向量数据库又是什么？

专门存储和检索"向量"（文本/图片的数学表示）的数据库。

**通俗理解**：
- 传统数据库：`SELECT * FROM articles WHERE content LIKE '%背单词%'`（精确匹配关键词）
- 向量数据库：把"背单词的技巧"变成一串数字（向量），找到内容含义最接近的其他向量（语义匹配）

**常见的向量数据库**：
- 轻量级：ChromaDB（适合本项目的 Python 项目）
- 企业级：Milvus, Pinecone, Weaviate

### 2.3 大模型 API

通过 HTTP 请求调用第三方 AI 模型，不需要自己训练。

| 服务商 | 价格 | 说明 |
|:---|:---|:---|
| **DeepSeek** | 极低（约 0.1 元/百万字） | 推荐，注册送额度 |
| **通义千问** | 免费额度大 | 阿里云，兼容 OpenAI 格式 |
| **OpenAI** | 较贵 | 需海外支付方式 |

### 2.4 本项目中用到的技术与 RAG 的关系总结

```
HabitFlow 现有技术栈
    ├── FastAPI (后端框架)
    ├── SQLAlchemy (ORM)
    ├── MySQL (关系型数据库)
    ├── Vue 3 (前端框架)
    ├── ECharts (图表)
    └── Element Plus (UI)

扩展时可能新增的技术
    ├── 内容库推送 → 不需要新技术，直接维护 Python 字典/JSON
    ├── 打卡模板    → 不需要新技术，扩展数据库字段
    ├── AI 建议     → 新增：requests 库调大模型 API
    ├── RAG 知识库  → 新增：ChromaDB + 文本嵌入模型 + LLM API
    ├── 数据导出   → 不需要新技术，Python 生成 CSV/PDF
    └── 游戏化     → 不需要新技术，扩展数据库 + 逻辑
```

## 三、功能一：内容推荐系统（按目标类型推送激励/知识内容）

### 3.1 功能描述

用户在完成打卡后，系统根据该目标所属的类型（英语、运动、阅读、学习等），返回一条与该类型相关的精选内容——包括名人名言、学习技巧、好词好句等。

同时，对于英语这类语言学习目标，还可以推送"今日词汇""每日一句"等知识类内容。

### 3.2 前后端交互流程

```
┌─────────────┐          ┌─────────────┐          ┌─────────────┐
│  前端页面    │          │  后端服务    │          │   内容库     │
│  (Vue 3)    │          │  (FastAPI)  │          │ (Python字典) │
└──────┬──────┘          └──────┬──────┘          └──────┬──────┘
       │                        │                        │
       │ 1. POST /api/check-ins │                        │
       │   (goalId, remark)     │                        │
       │───────────────────────►│                        │
       │                        │ 2. 保存打卡记录         │
       │                        │    (MySQL)             │
       │                        │───────────────────────►│
       │                        │ 3. get_inspiration()    │
       │                        │    (按 goal.type 随机取) │
       │                        │───────────────────────►│
       │                        │◄───────────────────────│
       │                        │  返回推荐内容            │
       │ 4. 返回 checkIn +      │                        │
       │    inspiration +       │                        │
       │    peerTips            │                        │
       │◄───────────────────────│                        │
       │                        │                        │
       │ 5. 显示打卡成功弹窗     │                        │
       │    展示推荐内容         │                        │
       │ 6. 用户点"继续加油"     │                        │
       │    弹窗关闭             │                        │
```

### 3.3 涉及的技术知识

| 知识点 | 在本功能中的应用 |
|:---|:---|
| **Python 数据结构** | 用字典（dict）按目标类型分类存储内容库，key 是类型字符串，value 是列表 |
| **随机算法** | `random.choice()` 从列表中随机选取一条内容 |
| **RESTful API 设计** | 在打卡 POST 接口的返回体中附加 `inspiration` 和 `peerTips` 字段 |
| **前端组件交互** | 使用 Element Plus 的 `ElMessageBox.alert()` 弹窗展示推荐内容 |
| **Axios 请求拦截** | 前端拿到扩展后的返回体，解析新增字段 |

### 3.4 关键代码结构

```
backend_fastapi/app/inspirations.py    ← 内容库 + 获取方法（纯数据，约50行）
backend_fastapi/app/routers/checkins.py ← 打卡接口中调用获取方法，附加到返回中
frontend/src/App.vue                    ← 前端打卡成功后解析返回，弹窗展示
```

### 3.5 这不是 RAG

这个功能**不是 RAG**。它只是从预定义的内容库中按**精确匹配**（通过 goal.type 匹配）随机抽取。没有用到向量检索，也没有用到大模型。相当于一个"分类字典 + 随机选择"。

## 四、功能二：打卡内容深度化（结构化打卡模板）

### 4.1 功能描述

当前打卡只允许填写一段备注文字，过于简单。可以针对不同类型的目标，提供结构化的打卡模板。

**样例对比：**

```
当前打卡（英语目标）：
  ┌─────────────────────────────┐
  │ 备注：今天背了50个单词       │
  └─────────────────────────────┘

增强后的打卡（英语目标）：
  ┌─────────────────────────────┐
  │ 📚 今日学习内容              │
  │   □ 背单词    数量：___ 个  │
  │   □ 练听力    时长：___ 分钟 │
  │   □ 读文章    篇数：___ 篇  │
  │   □ 练口语    时长：___ 分钟 │
  │ 📝 学习心得：______________  │
  │ 😊 今日状态：⭐⭐⭐⭐         │
  └─────────────────────────────┘
```

### 4.2 前后端交互流程

```
┌─────────────┐          ┌─────────────┐          ┌─────────────┐
│  前端        │          │  后端        │          │   数据库     │
└──────┬──────┘          └──────┬──────┘          └──────┬──────┘
       │                        │                        │
       │ 1. 用户选择目标         │                        │
       │ 2. 前端根据 goal.type   │                        │
       │    展示对应的打卡模板    │                        │
       │                        │                        │
       │ 3. POST /api/check-ins  │                        │
       │    { goalId, details:  │                        │
       │      { wordCount: 50,  │                        │
       │        listenMin: 30,  │                        │
       │        mood: 4 } }     │                        │
       │───────────────────────►│                        │
       │                        │ 4. 保存到 check_in     │
       │                        │    表新增的 details     │
       │                        │    字段 (JSON格式)      │
       │                        │───────────────────────►│
       │◄───────────────────────│                        │
       │ 5. 打卡成功 + 返回明细  │                        │
```

### 4.3 涉及的技术知识

| 知识点 | 在本功能中的应用 |
|:---|:---|
| **JSON 字段存储** | MySQL 的 `JSON` 类型或 VARCHAR 存序列化 JSON，存储结构化打卡数据 |
| **数据库表扩展** | `check_in` 表新增 `details JSON` 字段，或新建 `check_in_detail` 子表 |
| **前端模板渲染** | 按 `goal.type` 动态渲染不同的表单模板（英语模板/运动模板/阅读模板） |
| **Vue 条件渲染** | `v-if="goalType === '英语'"` 展示不同模板 |
| **数据校验** | 后端 Pydantic schema 扩展 `CheckInDetail` 模型，对不同类型的模板字段做校验 |

### 4.4 扩展 `check_in` 表

```sql
ALTER TABLE check_in ADD COLUMN details JSON DEFAULT NULL COMMENT '结构化打卡详情';
-- 示例 JSON:
-- {"wordCount": 50, "listenMin": 30, "mood": 4, "summary": "今天感觉不错"}
```

### 4.5 这类功能不算 RAG

仍然是纯关系数据库 + 前端模板渲染，没有检索增强生成。

## 五、功能三：成长日志与时间线

### 5.1 功能描述

在数据概览页或新增一个"成长日志"页面，按时间线展示用户的打卡历程，将每次打卡记录、系统推荐的激励语、获得的勋章整合成一条条"成长事件"。

**展示效果：**

```
2026-06-18  ─────────────────────────────────
  │ ✅ 在「每天背50个英语单词」上打卡了
  │ 💬 "The only limit... — 今日激励"
  │ 🏅 获得「连续7天勋章」！
  │ 📝 用户备注：今天背了60个，超额完成

2026-06-17  ─────────────────────────────────
  │ ✅ 在「每天背50个英语单词」上打卡了
  │ 📝 用户备注：复习了前几天的单词

2026-06-16  ─────────────────────────────────
  │ ✅ 在「每天运动30分钟」上打卡了
  │ 💬 "Take care of your body... — 运动语录"
```

### 5.2 前后端交互流程

```
┌─────────────┐          ┌─────────────┐          ┌─────────────┐
│  前端        │          │  后端        │          │   数据库     │
└──────┬──────┘          └──────┬──────┘          └──────┬──────┘
       │                        │                        │
       │ GET /api/timeline       │                        │
       │───────────────────────►│                        │
       │                        │ 查询 check_in 表       │
       │                        │ 查询 user_badge 表     │
       │                        │（按日期合并排序）       │
       │                        │───────────────────────►│
       │◄───────────────────────│                        │
       │ 返回合并后的时间线数据   │                        │
       │                        │                        │
       │ 前端用时间线组件渲染      │                        │
       │ Element Plus Timeline   │                        │
```

### 5.3 涉及的技术知识

| 知识点 | 在本功能中的应用 |
|:---|:---|
| **SQL 联表查询** | 分别查询 `check_in` 和 `user_badge`，在 Python 层按日期合并排序 |
| **Python 列表排序** | 使用 `sorted(events, key=lambda e: e['date'], reverse=True)` 合并排序 |
| **前端时间线组件** | Element Plus 的 `<el-timeline>` 组件 |
| **数据聚合** | 将打卡记录、勋章获得、系统推荐按时间轴归并成统一的事件流 |

### 5.4 核心算法思路

```python
def build_timeline(db, user_id):
    # 1. 查询所有打卡记录
    checkins = db.query(CheckIn).filter(...).all()
    # 2. 查询所有获得的勋章
    badges = db.query(UserBadge).join(Badge).filter(...).all()
    # 3. 合并成统一事件列表
    events = []
    for c in checkins:
        events.append({"date": c.check_date, "type": "checkin", "data": {...}})
    for b in badges:
        events.append({"date": b.obtained_time.date(), "type": "badge", "data": {...}})
    # 4. 按日期倒序排序
    events.sort(key=lambda e: e["date"], reverse=True)
    return events
```

### 5.5 与 RAG 的关系

不涉及 RAG。仍然是传统的关系数据库查询 + 内存排序。

## 六、功能四：AI 个性化建议（接入大模型 API）

### 6.1 功能描述

这是你提到的"打卡后给我提建议"的最高级实现方式。每次打卡完成后，系统将用户的打卡记录（目标类型、打卡备注、历史完成情况）发送给大模型 API，由 AI 生成个性化的鼓励和建议。

**场景示例：**

```
你创建了"每天背英语单词"的目标
今天打卡备注："今天背了50个单词，但感觉记不太住"

AI 回复：
"建议试试艾宾浩斯遗忘曲线复习法：
第1天、第2天、第4天、第7天、第15天分别复习一次，
记忆效率会提升3倍以上。明天可以先复习今天的50个，
再背30个新的。加油！"
```

### 6.2 前后端交互流程

```
┌─────────────┐          ┌─────────────┐          ┌─────────────┐          ┌─────────────┐
│  前端        │          │  后端        │          │   数据库     │          │  大模型 API  │
│  (Vue 3)    │          │  (FastAPI)  │          │   (MySQL)   │          │  (DeepSeek) │
└──────┬──────┘          └──────┬──────┘          └──────┬──────┘          └──────┬──────┘
       │                        │                        │                        │
       │ POST /api/check-ins    │                        │                        │
       │───────────────────────►│                        │                        │
       │                        │ 保存打卡记录            │                        │
       │                        │───────────────────────►│                        │
       │                        │ 查询历史数据            │                        │
       │                        │◄───────────────────────│                        │
       │                        │                        │                        │
       │                        │ 组装 Prompt:           │                        │
       │                        │ "用户正在坚持英语目标，  │                        │
       │                        │  打卡备注是背了50个     │                        │
       │                        │  单词但记不住。近7天    │                        │
       │                        │  连续打卡。请给建议。"  │                        │
       │                        │──────────────────────────────────────────────►│
       │                        │                        │                        │
       │                        │◄──────────────────────────────────────────────│
       │                        │  返回 AI 生成建议                                │
       │                        │                        │                        │
       │◄───────────────────────│                        │                        │
       │ 返回 checkIn +         │                        │                        │

       │                        │                        │                        │
       │ 前端弹窗显示 AI 建议     │                        │                        │
```

### 6.3 涉及的技术知识

| 知识点 | 在本功能中的应用 |
|:---|:---|
| **HTTP 客户端调用** | Python 的 `requests` 或 `httpx` 库调用大模型 API |
| **Prompt Engineering** | 设计提示词模板，告诉 AI 用户的上下文和期望的输出格式 |
| **环境变量/配置** | API Key 通过环境变量注入，不硬编码在代码中 |
| **错误处理** | API 调用失败时降级为本地推荐（功能一），保证打卡功能不受影响 |
| **速率限制** | 控制调用频率，避免短时间内大量请求导致费用超支 |
| **敏感信息过滤** | 发送给 AI 的内容不应包含密码等敏感信息 |

### 6.4 Prompt 设计模板

```python
def build_prompt(goal_type, goal_name, remark, streak_days, completion_rate):
    return f"""
你是一个自律目标助手的 AI 顾问。用户正在使用一个目标追踪平台。

用户的目标信息：
- 目标类型：{goal_type}
- 目标名称：{goal_name}
- 今日打卡备注：{remark or "无"}
- 连续打卡天数：{streak_days}天
- 目标完成率：{completion_rate}%

请根据以上信息，用中文给出一条具体的、可操作的建议或鼓励。
要求：
1. 不超过 80 个字
2. 贴近用户的目标类型（{goal_type}）
3. 如果用户有具体记录，给出针对性建议
4. 语气积极、鼓励性
"""
```

### 6.5 这是 RAG 吗？

**不是 RAG。** 这是**直接调用大模型 API（LLM API）**。区别在于：
- 我们只传了用户的个人数据，没有从外部知识库检索资料
- 回答完全依赖大模型自身的训练知识
- **RAG 需要额外增加检索步骤**，参见下一节

### 6.6 能不能说这是 RAG？

**严格来说不能。** 但答辩时你可以这样说：

> "我们实现了基于大模型的个性化建议功能，并预留了 RAG 架构的扩展接口。后续可以将高完成率用户的打卡经验整理成知识库，让 AI 基于真实成功案例给出更可靠的建议。"

这样既诚实，又显得你有前瞻性。

### 7.4 涉及的技术知识

| 知识点 | 在本功能中的应用 | 重要程度 |
|:---|:---|:---:|
| **文本嵌入（Embedding）** | 用 `text2vec-base-chinese` 等模型将文本转为向量，将语义相近的内容映射到向量空间中相近的位置 | ⭐⭐⭐⭐⭐ |
| **向量数据库（ChromaDB）** | 轻量级 Python 向量数据库，无需单独部署，`pip install chromadb` 即可用 | ⭐⭐⭐⭐⭐ |
| **语义搜索** | 不依赖关键词匹配，而是找到"意思最接近"的内容片段 | ⭐⭐⭐⭐ |
| **文本分块策略** | 将长文档切成合适大小的块（如 500 字/块），并保留重叠区域，避免上下文断裂 | ⭐⭐⭐ |
| **Prompt Engineering** | 将检索到的内容与用户问题组合成结构化提示词 | ⭐⭐⭐⭐ |
| **LLM API 调用** | 调用 DeepSeek/通义千问 生成最终回答 | ⭐⭐⭐⭐ |
| **引用溯源** | 回答中标注"参考了 XXX 用户的打卡经验"，增强可信度 | ⭐⭐⭐ |

### 7.6 这就是 RAG

这个功能**是真正的 RAG**。它包含了 RAG 的全部三个核心步骤：
1. **R**etrieval（检索）：从 ChromaDB 中找出相关知识
2. **A**ugmented（增强）：将检索结果附加到 Prompt 中
3. **G**eneration（生成）：大模型基于参考内容生成回答

### 7.7 RAG 的难点

| 难点 | 说明 | 解决方案 |
|:---|:---|:---|
| **知识库内容来源** | 需要整理高质量的学习资料 | 先收录系统内部高完成率用户的打卡经验（脱敏），后期再扩展外部资源 |
| **嵌入模型选择** | 中文嵌入质量影响检索效果 | 使用 `shibing624/text2vec-base-chinese`（开源，效果好） |
| **检索质量** | 可能搜到不相关的内容 | 设置相似度阈值 < 0.7 的结果不采用 |
| **响应速度** | 检索 + API 调用耗时可能 > 3s | 使用异步调用，前端先返回打卡结果，AI 建议异步加载 |

## 八、功能六：智能排行榜与推荐

### 8.1 功能描述

在现有的排行榜基础上，增加"智能推荐"维度：
- **推荐目标类型**：根据用户已创建的目标类型，推荐系统中同类目标完成率最高的用户的"打卡模式"
- **最佳打卡时间分析**：统计该用户历史打卡时间分布，推荐最佳打卡时间段
- **目标调整建议**：当用户某个目标连续多天未打卡时，自动建议缩短周期或降低每日次数

### 8.2 技术原理

| 技术 | 应用 | 说明 |
|:---|:---|:---|
| **统计分析** | 聚合查询计算不同模式下的平均完成率 | `GROUP BY cycle, daily_target_count` |
| **协同过滤思想** | "和你类似目标的用户，他们的做法是…" | 不涉及矩阵分解，只是分组统计 |
| **时间序列分析** | 分析用户每日打卡时间，找出规律 | 对 `check_time` 做直方图统计 |

### 8.3 与 RAG 的关系

不涉及 RAG。主要是 SQL 聚合统计 + 简单数据分析。

```python
"""数据导出服务"""

import io
from datetime import date
from openpyxl import Workbook
from openpyxl.styles import Font, PatternFill, Alignment, Border, Side
from fastapi.responses import StreamingResponse
from sqlalchemy.orm import Session


def export_checkins_to_excel(db: Session, user_id: int, 
                              start_date: date = None, 
                              end_date: date = None) -> StreamingResponse:
    """将打卡记录导出为 Excel 文件"""
    
    # 查询数据
    from .models import CheckIn, Goal
    query = db.query(CheckIn, Goal.name, Goal.type).join(
        Goal, CheckIn.goal_id == Goal.id
    ).filter(CheckIn.user_id == user_id)
    
    if start_date:
        query = query.filter(CheckIn.check_date >= start_date)
    if end_date:
        query = query.filter(CheckIn.check_date <= end_date)
    
    rows = query.order_by(CheckIn.check_date.desc()).all()
    
    # 创建 Excel 工作簿
    wb = Workbook()
    ws = wb.active
    ws.title = "打卡记录"
    
    # 设置样式
    header_font = Font(name='微软雅黑', bold=True, color='FFFFFF', size=11)
    header_fill = PatternFill(start_color='2F80ED', end_color='2F80ED', fill_type='solid')
    thin_border = Border(
        left=Side(style='thin'),
        right=Side(style='thin'),
        top=Side(style='thin'),
        bottom=Side(style='thin')
    )
    
    # 表头
    headers = ['日期', '目标名称', '目标类型', '备注内容', '打卡类型', '打卡时间']
    for col, header in enumerate(headers, 1):
        cell = ws.cell(row=1, column=col, value=header)
        cell.font = header_font
        cell.fill = header_fill
        cell.border = thin_border
        cell.alignment = Alignment(horizontal='center')
    
    # 数据行
    for row_idx, (checkin, goal_name, goal_type) in enumerate(rows, 2):
        ws.cell(row=row_idx, column=1, value=checkin.check_date.isoformat()).border = thin_border
        ws.cell(row=row_idx, column=2, value=goal_name).border = thin_border
        ws.cell(row=row_idx, column=3, value=goal_type).border = thin_border
        ws.cell(row=row_idx, column=4, value=checkin.remark or '').border = thin_border
        ws.cell(row=row_idx, column=5, value='补卡' if checkin.makeup else '正常打卡').border = thin_border
        ws.cell(row=row_idx, column=6, value=checkin.check_time.strftime('%Y-%m-%d %H:%M')).border = thin_border
    
    # 设置列宽
    ws.column_dimensions['A'].width = 14
    ws.column_dimensions['B'].width = 22
    ws.column_dimensions['C'].width = 12
    ws.column_dimensions['D'].width = 40
    ws.column_dimensions['E'].width = 12
    ws.column_dimensions['F'].width = 18
    
    # 写入内存流
    output = io.BytesIO()
    wb.save(output)
    output.seek(0)
    
    # 返回文件下载响应
    filename = f"打卡记录_{date.today().isoformat()}.xlsx"
    return StreamingResponse(
        output,
        media_type="application/vnd.openxmlformats-officedocument.spreadsheetml.sheet",
        headers={"Content-Disposition": f"attachment; filename={filename}"}
    )
```

```python
    db: Session = Depends(get_db),
    current_user: User = Depends(get_current_user),
):
    """导出打卡记录"""
    if format == "xlsx":
        return export_checkins_to_excel(db, current_user.id, start_date, end_date)
    return {"code": 400, "message": "不支持的导出格式", "data": None}
```

### 10.6 关键技术点解析

**Q：把文件写入内存而不写入磁盘，有什么好处？**
A：写入 BytesIO 内存流后直接返回给前端，不在服务器上留下临时文件。优点是：无磁盘 I/O 延迟、不占用磁盘空间、不会有垃圾文件残留。

**Q：哪里需要装新依赖？**
A：在项目根目录执行 `pip install openpyxl` 即可，openpyxl 是纯 Python 的 Excel 库，不需要额外安装 Office 软件。

**Q：PDF 导出怎么实现？**
A：可以使用 `reportlab` 库生成 PDF，或者在后端先生成 Excel，再用 `libreoffice --headless --convert-to pdf` 转换。对于课程设计，导出 Excel 已经足够体现能力了。

### 11.1 功能描述

### 11.6 核心计算逻辑

```python
def calculate_leaderboard(db: Session, challenge_id: int) -> list[dict]:
    """计算挑战排行榜"""
    from .models import Challenge, ChallengeParticipant, CheckIn, Goal
    
    challenge = db.query(Challenge).filter(Challenge.id == challenge_id).first()
    if not challenge:
        return []
    
    participants = db.query(ChallengeParticipant).filter(
        ChallengeParticipant.challenge_id == challenge_id
    ).all()
    
    result = []
    for participant in participants:
        # 查询该用户在挑战期间的打卡数据
        query = db.query(CheckIn).filter(
            CheckIn.user_id == participant.user_id,
            CheckIn.check_date >= challenge.start_date,
            CheckIn.check_date <= challenge.end_date,
            CheckIn.status == "DONE"
        )
        
        # 如果挑战限制了目标类型，只统计该类型的打卡
        if challenge.goal_type:
            query = query.join(Goal).filter(Goal.type == challenge.goal_type)
        
        checkins = query.all()
        
        # 根据考核指标计算分数
        if challenge.metric == "checkin_count":
            score = len(checkins)
        elif challenge.metric == "streak":
            # 计算挑战期间的最长连续天数
            dates = sorted(set(c.check_date for c in checkins))
            max_streak = 0
            current_streak = 0
            for i, d in enumerate(dates):
                if i == 0 or (d - dates[i-1]).days == 1:
                    current_streak += 1
                else:
                    max_streak = max(max_streak, current_streak)
                    current_streak = 1
            max_streak = max(max_streak, current_streak)
            score = max_streak
        else:
            score = 0
        
        result.append({
            "userId": participant.user_id,
            "score": score,
            "joinTime": participant.join_time,
        })
    
    # 按分数降序排列
    result.sort(key=lambda x: x["score"], reverse=True)
    
    # 标记排名
    for rank, item in enumerate(result, 1):
        item["rank"] = rank
    
    return result
```
