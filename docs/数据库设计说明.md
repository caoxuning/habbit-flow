# HabitFlow 数据库设计说明

## 设计目标

HabitFlow 的数据库围绕“用户创建目标、每日打卡、社交监督、消息提醒、勋章激励”这条核心链路设计。表结构来自当前远端 RDS，并已同步到 `database/schema.sql`。

数据库采用 MySQL 8.0，字符集为 `utf8mb4`，可以保存中文目标、评论、私信和圈子内容。主键统一使用 `bigint auto_increment`，用户、目标、打卡、社交内容之间通过外键保持关联。

## 模块划分

### 用户与账号模块

核心表：`user`

`user` 保存系统用户的基础信息，包括用户名、加密密码、邮箱和创建时间。用户名同时用于登录、好友列表、圈子动态、评论和聊天展示。

重要字段：

- `id`：用户唯一编号，其他表通过它关联用户。
- `username`：用户名/展示名，具有唯一约束。
- `password`：加密后的密码，不保存明文。
- `email`：邮箱，用于个人资料展示。
- `create_time`：账号创建时间。

### 目标与打卡模块

核心表：`goal`、`check_in`

`goal` 保存用户创建的习惯目标，例如“力量训练 45 分钟”“英语听力精听 20 分钟”。目标包含周期、开始结束日期、每日目标次数、优先级和状态。

`check_in` 保存用户对目标的打卡记录。每条记录对应一个目标和一个用户，记录打卡日期、提交时间、备注和是否补卡。

主要关系：

- `goal.user_id` -> `user.id`
- `check_in.goal_id` -> `goal.id`
- `check_in.user_id` -> `user.id`

这两张表支撑工作台、任务日历、打卡记录、成长日志、排行榜和勋章统计。

### 社交圈子模块

核心表：`social_circle`、`circle_member`、`circle_post`、`post_comment`、`post_like`

`social_circle` 保存圈子信息，例如运动打卡圈、英语学习圈。
`circle_member` 保存用户加入圈子的关系。
`circle_post` 保存圈子动态，包括普通动态和打卡分享。
`post_comment` 保存动态评论。
`post_like` 保存动态点赞。

主要关系：

- `social_circle.owner_user_id` -> `user.id`
- `circle_member.circle_id` -> `social_circle.id`
- `circle_member.user_id` -> `user.id`
- `circle_post.circle_id` -> `social_circle.id`
- `circle_post.user_id` -> `user.id`
- `circle_post.check_in_id` -> `check_in.id`
- `post_comment.post_id` -> `circle_post.id`
- `post_like.post_id` -> `circle_post.id`

该模块用于圈子广场、社区动态、评论互动、打卡分享和圈子排行榜。

### 好友与私信模块

核心表：`friendship`、`direct_message`

`friendship` 同时保存好友申请和好友关系状态。申请发起人为 `requester_id`，接收人为 `addressee_id`，状态可表示待处理、已同意或已拒绝。

`direct_message` 保存好友之间的一对一聊天消息。每条消息记录发送者、接收者、内容和发送时间。

主要关系：

- `friendship.requester_id` -> `user.id`
- `friendship.addressee_id` -> `user.id`
- `direct_message.sender_id` -> `user.id`
- `direct_message.receiver_id` -> `user.id`

该模块用于好友管理、好友申请、好友今日打卡看板和聊天功能。

### 消息提醒模块

核心表：`notification`

`notification` 保存已经写入数据库的提醒消息，包括每日打卡提醒、目标到期预警和连续中断提醒。提醒列表接口只读取该表，不在查询时自动创建新提醒。

重要字段：

- `user_id`：提醒接收用户。
- `type`：提醒类型，例如 `DAILY_CHECK_IN`、`GOAL_EXPIRE`、`STREAK_BREAK`。
- `title`：提醒标题。
- `content`：提醒正文。
- `is_read`：是否已读。
- `create_time`：提醒生成时间。

该表支撑消息提醒页面、顶部未读数和一键已读功能。

### 勋章激励模块

核心表：`badge`、`user_badge`

`badge` 保存系统预设的勋章定义，包括编码、名称、说明和解锁条件文案。
`user_badge` 保存用户已经获得的勋章记录。

主要关系：

- `user_badge.user_id` -> `user.id`
- `user_badge.badge_id` -> `badge.id`

该模块用于勋章墙、成长激励和连续打卡成就展示。

## 核心业务链路

### 目标打卡链路

1. 用户登录后读取 `user`。
2. 用户在目标管理中创建目标，写入 `goal`。
3. 用户每天完成目标后提交打卡，写入 `check_in`。
4. 工作台、日历、成长日志从 `goal` 和 `check_in` 汇总数据。
5. 系统根据打卡次数和连续情况判断是否写入 `user_badge`。

### 社交监督链路

1. 用户通过搜索同学账号发起好友申请，写入 `friendship`。
2. 对方同意后，`friendship.status` 变为已接受。
3. 好友之间可以查看今日打卡状态，数据来自 `check_in`。
4. 好友可以通过 `direct_message` 发送聊天消息或打卡提醒。

### 圈子互动链路

1. 用户创建或加入圈子，写入 `social_circle` 和 `circle_member`。
2. 用户发布动态或分享打卡，写入 `circle_post`。
3. 其他用户评论或点赞，写入 `post_comment` 和 `post_like`。
4. 社区动态页面汇总用户已加入圈子的内容。

### 消息提醒链路

1. 业务流程或后续定时任务明确写入提醒到 `notification`。
2. 前端读取未读提醒并展示在顶部导航和消息页面。
3. 用户进入消息页面或处理提醒后，更新 `notification.is_read`。

## 表关系简图

```text
user
  ├─ goal
  │   └─ check_in
  │       └─ circle_post
  ├─ friendship
  ├─ direct_message
  ├─ social_circle
  │   └─ circle_member
  ├─ circle_post
  │   ├─ post_comment
  │   └─ post_like
  ├─ notification
  └─ user_badge
      └─ badge
```

## 中期展示关注点

展示时可以重点说明三点：

1. 数据持久化：所有用户、目标、打卡、好友、圈子和消息都写入远端 RDS。
2. 核心链路闭环：目标创建 -> 每日打卡 -> 统计成长 -> 社交监督 -> 勋章激励。
3. 多人协作：不同成员本地启动后端时，只要连接同一个 RDS，就能访问同一份数据。
