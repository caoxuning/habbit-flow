# HabitFlow 核心链路与测试标准

本文用于说明 HabitFlow 项目的核心业务链路、关键卡点、API 优先验证方式、成功标准和测试流程。建议组内开发、联调、答辩演示都按本文档执行。

## 1. 核心链路

### 1.1 第一核心链路：个人自律闭环

个人自律闭环是项目最重要的主线，也是答辩时必须优先保证稳定演示的功能。

```text
注册/登录 -> 创建目标 -> 今日打卡/补卡 -> 查看统计 -> 自动获得勋章
```

这条链路证明系统不是单纯页面展示，而是具备完整业务能力：

- 用户可以创建账号并登录。
- 用户可以创建自己的自律目标。
- 用户可以对目标进行今日打卡。
- 系统可以拦截重复打卡和非法补卡。
- 打卡后统计数据会变化。
- 满足条件后系统会自动发放勋章。

### 1.2 第二核心链路：社交监督闭环

社交监督闭环是成员三负责的重点，用于体现 HabitFlow 从个人工具升级为社交平台。

好友链路：

```text
搜索用户 -> 发送好友申请 -> 对方同意 -> 查看好友列表
```

圈子链路：

```text
加入圈子 -> 发布帖子 -> 查看圈子帖子 -> 查看我的圈子动态流
```

这条链路证明系统具备基础互动能力：

- 用户之间可以建立好友关系。
- 用户可以加入兴趣圈子。
- 加入圈子后可以发布动态。
- 动态流可以汇总用户加入圈子的帖子。

## 2. 为什么优先用 API 验证

建议先用 API 验证核心链路，再做前端联调。

原因：

- API 能直接验证后端业务是否真实可用。
- API 能更容易定位问题是在前端、后端还是数据库。
- API 测试稳定后，前端只需要验证按钮和页面数据展示。
- 答辩时如果前端临时出现样式或缓存问题，API 仍可以证明核心业务成立。

推荐验证顺序：

```text
API 测试 -> 前端联调 -> 异常场景测试 -> 答辩演示
```

## 3. 关键卡点与成功标准

| 卡点 | 验证内容 | 成功标准 |
| --- | --- | --- |
| 登录认证 | 注册/登录后是否返回 token，后续接口是否能携带 token 访问 | 登录返回 `token`，带 token 可访问 `/api/user/profile` |
| 目标创建 | 目标名称、类型、日期、周期、每日次数是否保存 | `/api/goals` 能查到新目标 |
| 今日打卡 | 能否对进行中目标打卡 | `/api/check-ins` 返回新打卡记录 |
| 重复打卡 | 同一目标同一天不能重复打卡 | 第二次打卡返回业务错误 |
| 补卡校验 | 补卡日期必须早于今天，且在目标周期内 | 合法补卡成功，非法补卡失败 |
| 统计刷新 | 打卡后统计数据是否变化 | `totalCheckIns`、`completionRate` 等字段更新 |
| 勋章发放 | 首次打卡后是否自动获得勋章 | `/api/badges/mine` 返回“初次打卡勋章” |
| 用户搜索 | 登录用户能否搜索其他用户 | 搜索结果排除自己，返回目标用户 |
| 好友申请 | 不能添加自己，不能重复申请 | 正常申请成功，重复申请失败 |
| 好友通过 | 被申请人能否接受申请 | 接受后双方好友列表能看到对方 |
| 圈子加入 | 用户能否加入默认圈子 | 加入后 `joined` 为 `true`，成员数增加 |
| 圈子发帖 | 用户必须加入圈子才能发帖 | 未加入发帖失败，加入后发帖成功 |
| 动态流 | 用户能否看到已加入圈子的帖子 | `/api/social/feed` 返回圈内帖子 |

## 4. 环境准备

### 4.1 启动后端

```bash
cd backend_fastapi
py -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload --host 0.0.0.0 --port 8081
```

后端地址：

```text
http://localhost:8081
```

FastAPI 自动接口文档：

```text
http://localhost:8081/docs
```

### 4.2 启动前端

```bash
cd frontend
npm.cmd install
npm.cmd run dev
```

前端地址：

```text
http://localhost:5173
```

### 4.3 数据库要求

后端默认连接：

```text
mysql+pymysql://root:root@localhost:3306/habitflow?charset=utf8mb4
```

如果本地 MySQL 账号密码不同，需要设置环境变量：

```bash
set DATABASE_URL=mysql+pymysql://用户名:密码@localhost:3306/habitflow?charset=utf8mb4
```

## 5. API 测试流程

推荐在 FastAPI 自动接口文档中测试：

```text
http://localhost:8081/docs
```

### 5.1 个人自律闭环测试

#### 步骤 1：注册用户 A

接口：

```http
POST /api/auth/register
```

请求示例：

```json
{
  "username": "user_a",
  "password": "123456",
  "email": "user_a@example.com"
}
```

成功标准：

- 返回 `code = 200`。
- 返回 `token`。
- 返回 `profile`。

#### 步骤 2：登录用户 A

接口：

```http
POST /api/auth/login
```

请求示例：

```json
{
  "username": "user_a",
  "password": "123456"
}
```

成功标准：

- 返回 `token`。
- 后续接口可以使用该 token。

#### 步骤 3：设置认证 token

在 FastAPI `/docs` 页面点击右上角 `Authorize`，填写：

```text
Bearer 登录返回的token
```

成功标准：

- 调用需要登录的接口不会返回 401。

#### 步骤 4：创建目标

接口：

```http
POST /api/goals
```

请求示例：

```json
{
  "name": "每天阅读",
  "type": "阅读",
  "startDate": "2026-06-01",
  "endDate": "2026-06-30",
  "cycle": "DAILY",
  "dailyTargetCount": 1,
  "status": "ACTIVE"
}
```

成功标准：

- 返回新目标 `id`。
- 返回目标状态为 `ACTIVE`。

#### 步骤 5：查询目标列表

接口：

```http
GET /api/goals
```

成功标准：

- 返回刚创建的目标。
- 返回 `completedCount`、`expectedCount`、`completionRate`。

#### 步骤 6：今日打卡

接口：

```http
POST /api/check-ins
```

请求示例：

```json
{
  "goalId": 1,
  "remark": "完成阅读 30 分钟"
}
```

成功标准：

- 返回打卡记录。
- `makeup` 为 `false`。
- `status` 为 `DONE`。

#### 步骤 7：重复打卡校验

再次调用：

```http
POST /api/check-ins
```

请求体和上一步相同。

成功标准：

- 返回业务错误。
- 错误信息类似：`该目标当天已打卡`。

#### 步骤 8：补卡

接口：

```http
POST /api/check-ins/makeup
```

请求示例：

```json
{
  "goalId": 1,
  "checkDate": "2026-06-17",
  "remark": "昨天忘记提交，实际已完成"
}
```

成功标准：

- 合法补卡返回成功。
- `makeup` 为 `true`。
- 如果补卡日期不是过去日期，应返回业务错误。

#### 步骤 9：查看统计

接口：

```http
GET /api/stats/dashboard
```

成功标准：

- `totalGoals` 大于等于 1。
- `totalCheckIns` 大于等于 1。
- `averageCompletionRate` 有数值。
- `monthlyReport` 有最近月份数据。
- `goalProgress` 包含目标完成率。

#### 步骤 10：查看勋章

接口：

```http
GET /api/badges/mine
```

成功标准：

- 首次打卡后返回“初次打卡勋章”。
- 返回字段包含 `name`、`description`、`conditionText`、`obtainedTime`。

### 5.2 社交监督闭环测试

社交测试至少需要两个用户。

#### 步骤 1：注册用户 B

接口：

```http
POST /api/auth/register
```

请求示例：

```json
{
  "username": "user_b",
  "password": "123456",
  "email": "user_b@example.com"
}
```

成功标准：

- 用户 B 注册成功。

#### 步骤 2：用户 A 搜索用户 B

切换回用户 A 的 token。

接口：

```http
GET /api/social/users/search?keyword=user_b
```

成功标准：

- 返回用户 B。
- 搜索结果不包含用户 A 自己。
- `friendshipStatus` 初始为 `NONE`。

#### 步骤 3：用户 A 发送好友申请

接口：

```http
POST /api/social/friend-requests
```

请求示例：

```json
{
  "targetUserId": 2,
  "message": "一起坚持打卡吧"
}
```

成功标准：

- 返回好友申请记录。
- `status` 为 `PENDING`。

#### 步骤 4：用户 B 查看好友申请

切换为用户 B 的 token。

接口：

```http
GET /api/social/friend-requests
```

成功标准：

- 用户 B 能看到用户 A 发来的申请。
- 申请状态为 `PENDING`。

#### 步骤 5：用户 B 同意好友申请

接口：

```http
PUT /api/social/friend-requests/{requestId}/accept
```

成功标准：

- 返回成功。
- 好友申请状态变为 `ACCEPTED`。

#### 步骤 6：查看好友列表

接口：

```http
GET /api/social/friends
```

成功标准：

- 用户 A 的好友列表能看到用户 B。
- 用户 B 的好友列表能看到用户 A。

#### 步骤 7：查看圈子列表

接口：

```http
GET /api/social/circles
```

成功标准：

- 返回默认圈子，例如英语打卡圈、健身打卡圈、阅读打卡圈。
- 每个圈子包含 `id`、`name`、`description`、`memberCount`、`joined`。

#### 步骤 8：加入圈子

接口：

```http
POST /api/social/circles/{circleId}/join
```

成功标准：

- 返回成功。
- 再次查询圈子列表时，该圈子的 `joined` 为 `true`。

#### 步骤 9：发布圈子帖子

接口：

```http
POST /api/social/circles/{circleId}/posts
```

请求示例：

```json
{
  "content": "今日英语阅读 30 分钟，完成打卡。"
}
```

成功标准：

- 加入圈子后发帖成功。
- 未加入圈子时发帖失败。

#### 步骤 10：查看圈子帖子

接口：

```http
GET /api/social/circles/{circleId}/posts
```

成功标准：

- 返回刚发布的帖子。
- 帖子包含作者、圈子名称、内容和发布时间。

#### 步骤 11：查看我的圈子动态流

接口：

```http
GET /api/social/feed
```

成功标准：

- 返回当前用户已加入圈子的帖子。
- 按发布时间倒序展示。

## 6. 前端联调测试

API 测试通过后，再启动前端进行页面验证。

前端演示路径：

```text
登录/注册 -> 数据概览 -> 目标管理 -> 打卡记录 -> 勋章奖励 -> 社交圈子
```

个人功能页面成功标准：

- 登录后能进入主页面。
- 创建目标后目标列表刷新。
- 点击打卡后打卡记录增加。
- 数据概览指标变化。
- 勋章页面显示已获得勋章。

社交页面成功标准：

- “社交圈子”菜单可以进入。
- “好友管理”可以搜索用户、发送好友申请、处理好友申请、查看好友列表。
- “圈子广场”可以查看默认圈子、创建圈子、加入圈子、退出圈子。
- 加入圈子后可以发布动态。
- “社区动态”可以看到已加入圈子的帖子。

## 7. 异常测试清单

建议至少测试以下异常情况：

- 不带 token 调用需要登录的接口，应返回 401。
- 使用错误密码登录，应返回业务错误。
- 创建目标时结束日期早于开始日期，应失败。
- 对已暂停或已完成目标打卡，应失败。
- 同一目标同一天重复打卡，应失败。
- 补卡日期为今天或未来日期，应失败。
- 搜索用户时不应搜索到自己。
- 给自己发送好友申请，应失败。
- 对同一用户重复发送好友申请，应失败。
- 未加入圈子直接发帖，应失败。
- 圈主退出自己创建的圈子，应失败。

## 8. 答辩演示推荐路线

答辩时建议先演示最稳定、最能说明工作量的路线：

```text
注册登录
-> 创建目标
-> 今日打卡
-> 查看打卡记录
-> 查看统计图表
-> 查看勋章
-> 搜索用户并发送好友申请
-> 加入圈子
-> 发布圈子动态
-> 查看社区动态流
```

如果现场时间不够，可以只演示：

```text
登录 -> 创建目标 -> 打卡 -> 数据统计 -> 勋章 -> 圈子发帖
```

这条路线能覆盖个人闭环和社交亮点，适合作为最终展示主线。
